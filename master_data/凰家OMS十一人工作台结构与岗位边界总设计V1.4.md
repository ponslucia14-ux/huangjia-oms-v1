# 一、文档说明

## 1.1 文档目的

本文件用于统一确认凰家十一人在 OMS 中：

- 首页第一眼看什么；
- 每天真正需要完成什么；
- 可以录入、查看、核对、复核或审批什么；
- 操作后应产生什么系统结果；
- 与其他岗位如何传递数据；
- 哪些事情不属于本人，必须禁止；
- 手机端和飞书实机应如何验收。

本文件是后续工作台页面设计、代码开发和飞书实机验收的统一岗位依据。

## 1.2 V1 设计原则

```text
尽量少改现有工作方式
→ 去掉重复沟通和重复录入
→ 每个人只做自己真正负责的事
→ 同一业务事实只录一次
→ 系统自动计算和汇总
→ 先形成稳定业务闭环
→ 再根据真实使用进入 V2
```

必须遵守：

1. 一个岗位围绕一个核心目标设计；双岗位必须分区。
2. 已有数据自动带出，不让员工重复抄写。
3. 录入者不承担复核，复核者不重做原始录入。
4. 查看岗位不得出现写入按钮。
5. V1 只做当前业务必需的录入、核对、审批、查询和简单汇总。
6. 复杂分析、预测、推荐和深度自动化进入 V2。
7. 菜单存在不等于岗位逻辑正确，最终以飞书电脑端和手机端真实使用为准。

数据沉淀统一遵循：

> V1 负责自然沉淀数据，不额外增加员工录入负担；V2 再基于沉淀数据建设分析、预测和智能经营能力。

## 1.3 数据与显示原则

- 当前事实必须来自 OMS 中真实录入、确认和复核后的数据。
- 未审核数据可以显示，但必须标明“待确认”“待复核”等状态和数据截止时间。
- 统计结果由系统计算，不要求员工填写汇总数字。
- 同一业务事实只录入一次；后续岗位通过确认、复核、审批或只读方式使用，不得重新抄写。
- 员工完成原有业务动作时，系统在后台保存原始数据、必要分类、状态、来源和人工修改记录。
- 不得为建设数据池新增与当前工作无关的录入表单，也不得把 V2 分析能力包装成 V1 功能。
- 员工页面除“OMS”外全部使用中文。
- EMP001 员工可见名称统一显示“10晓磊”，底层保留正式姓名、真实飞书身份和 EMP 关系。
- 陈晶辉和周志朋第一版为纯只读工作台。

# 四、十一人独立工作台设计

# 4.1 EMP001 10晓磊｜主理办、销售总监、资金决策责任

## 1. 岗位定位

10晓磊是经营决策、审批和跨岗位例外裁决的最终责任人。工作台不是各部门录入页面的合集，只负责看全局、看风险、处理必须本人决定的事项并追溯执行结果。

## 2. 唯一核心目标

```text
快速看清经营现状
→ 处理必须本人决定的事项
→ 查看执行结果
```

## 3. 首页第一眼看什么

- 今日必须处理；
- 经营快照；
- 销售变化；
- 资金状态；
- 在住与房态；
- 风险提醒；
- 数据截止时间、审核状态和结单状态。

未审核数据可显示，但必须明确标注，例如：

```text
今日新增签约 1 单｜待财务确认
```

## 4. 一级菜单

```text
首页
待我决策
经营状态
风险异常
决策追溯
```

## 5. 可查看内容

- 三名销售分别维护后汇总的销售状态；
- 刘晶录入、张敬东复核后的资金状态；
- 刘芳羽确认的当前在住、房态和排房；
- 待审批、待授权、超时事项和跨岗位冲突；
- 本人的审批、授权和异常处理记录。

## 6. 可操作内容

- 查看；
- 审批或驳回；
- 要求补充并填写原因；
- 例外授权；
- 确认是否介入重大风险。

## 7. 不可操作内容

- 替销售录入合同或接待；
- 替刘晶录入收付款；
- 替张敬东复核；
- 替刘芳羽排房或发布房态；
- 替尚丽娜维护客户现场资料；
- V1 不建设发布经营指令和复杂线上指令管理。

## 8. 数据闭环

```text
责任岗位提交真实事项
→ 10晓磊查看依据和影响
→ 审批、驳回、要求补充或授权
→ 原责任岗位执行
→ 结果回写 OMS
→ 10晓磊追溯结果
```

各销售分别维护本人销售事实，系统汇总后由 10晓磊查看；销售归属、跨人员冲突或异常由 10晓磊裁决，不由杨欢欢统一确认。

## 9. 手机端重点

- 第一屏优先显示今日必须处理、经营快照和风险；
- 审批按钮足够大，审批与查看明确分离；
- 任何审批必须打开详情、填写原因后提交；
- 不显示销售、财务、运营的日常录入表单。

## 10. 飞书实机验收

```text
使用 EMP001 真实飞书身份登录
→ 查看首页经营状态和截止时间
→ 打开待我决策
→ 查看一项真实依据
→ 通过或退回并填写原因
→ 查看决策追溯
→ 返回首页
```

## 11. V2 预留

- 审批金额阈值；
- 例外授权期限；
- 复杂经营分析；
- 自动风险预测。

# 4.2 EMP002 宗惠｜人事

## 1. 岗位定位

V1 不建设完整人力资源系统。宗惠只围绕正式员工基础资料、钉钉考勤和工资候选开展工作。

## 2. 唯一核心目标

```text
导入考勤
→ OMS 自动算薪
→ 核对异常
→ 提交工资
```

## 3. 首页第一眼看什么

- 本月钉钉考勤是否上传；
- 哪些员工考勤异常；
- 哪些奖惩待确认；
- 工资候选是否生成；
- 待提交工资；
- 审批、付款和核算状态。

## 4. 一级菜单

```text
首页
员工与薪资基础
钉钉考勤导入
考勤异常
工资候选
提交与结果
```

## 5. 可查看内容

- 正式员工花名册、岗位和状态；
- 当前及历史薪资标准；
- 考勤导入结果和异常；
- 工资候选、审批、付款和核算结果。

## 6. 可操作内容

- 维护员工花名册；
- 新增、离职、转岗、晋升和降职；
- 维护涨薪、降薪、薪资标准和奖惩规则；
- 上传钉钉考勤文件；
- 核对异常；
- 提交工资候选。

## 7. 不可操作内容

- 培训管理；
- 排班计划和缺班管理；
- 招聘流程；
- 复杂绩效系统；
- 执行工资付款或会计核算；
- 逐人重新录入钉钉已有考勤。

## 8. 数据闭环

```text
宗惠维护正式员工基础数据
→ 每月上传钉钉考勤
→ OMS 自动计算考勤和工资
→ 宗惠核对异常并提交
→ 10晓磊审批
→ 刘晶付款
→ 张敬东核算
→ 宗惠查看结果
```

## 9. 手机端重点

- 手机端只查看异常、补充必要说明、提交和查看结果；
- 考勤文件上传和批量核对优先电脑端；
- 被退回后直接显示缺什么并进入补充位置。

## 10. 飞书实机验收

```text
EMP002 真实身份登录
→ 查看考勤上传状态
→ 打开一项考勤异常
→ 补充说明并保存
→ 查看工资候选
→ 提交审批
→ 查看后续状态
```

## 11. V2 预留

复杂薪酬模型、绩效、招聘、培训和员工发展。

# 4.3 EMP003 张敬东｜会计

## 1. 岗位定位

张敬东是财务专业复核、会计分类、核算、对账和结算责任人，不创建刘晶负责的原始资金事实。

## 2. 唯一核心目标

```text
确认财务数据真实、完整、可入账
```

## 3. 首页第一眼看什么

- 待复核收款和付款；
- 凭证缺失与金额差异；
- 待收和待付；
- 待核算工资；
- 账户差异与当日结单状态。

## 4. 一级菜单

```text
首页
收款复核
付款复核
工资核算
对账与结算
财务追溯
```

## 5. 可查看内容

合同和付款依据、实际收付款、凭证、工资审批结果、账户余额、历史复核和更正记录。

## 6. 可操作内容

- 复核、确认或退回；
- 要求补充；
- 会计分类；
- 对账、核算和结算；
- 发起留痕更正；
- 查看凭证和历史。

## 7. 不可操作内容

- 代替刘晶创建实际收付款；
- 修改销售合同、入住或房态原始事实；
- 自己录入后自己复核；
- 删除已确认记录。

## 8. 数据闭环

```text
客户收款：
销售录入合同与付款信息
→ 刘晶确认并录入真实到账
→ 张敬东复核
→ 形成收款事实

对外付款、客户退款和工资：
业务责任人提交
→ 10晓磊审批
→ 刘晶执行付款并上传凭证
→ 张敬东复核或核算
→ 形成付款事实
```

## 9. 手机端重点

手机端突出待复核、差异和凭证详情；批量对账和复杂核算优先电脑端。

## 10. 飞书实机验收

```text
EMP003 真实身份登录
→ 查看待复核
→ 打开收付款及凭证
→ 通过或退回并填写原因
→ 查看对账结果
→ 返回首页
```

## 11. V2 预留

自动对账、差异识别、成本归集、财务预测和利润分析。

# 4.4 EMP004 刘晶｜出纳

## 1. 岗位定位

刘晶是所有真实资金动作的标准录入者，负责记录实际发生的收款、付款、凭证、账户余额和日结。

## 2. 唯一核心目标

```text
把真实发生的每一笔收款和付款准确录入
```

## 3. 首页第一眼看什么

- 今日收款和付款；
- 待收、待付；
- 待补凭证；
- 被退回记录；
- 账户余额和日结状态。

## 4. 一级菜单

```text
首页
收款
付款
工资付款
日结
我的记录
```

## 5. 可查看内容

付款依据、审批结果、本人录入记录、张敬东退回意见、账户余额和日结结果。

## 6. 可操作内容

- 录入实际到账和实际付款；
- 上传凭证；
- 保存草稿；
- 提交复核；
- 补充被退回资料；
- 录入账户余额并提交日结。

## 7. 不可操作内容

- 复核自己的记录；
- 修改合同金额；
- 把销售首笔付款信息当作真实到账；
- 绕过审批付款；
- 修改入住和房态；
- 删除已确认流水。

## 8. 数据闭环

```text
客户收款：
销售录入合同与付款信息
→ 刘晶确认实际到账并上传凭证
→ 张敬东复核
→ 形成收款事实

对外付款、客户退款、正式员工工资和照护师工资：
业务责任人提交
→ 10晓磊审批
→ 刘晶执行付款并上传凭证
→ 张敬东复核或核算
→ 形成付款事实
```

V1 暂不设置金额分级阈值。所有上述付款均执行同一审批链，不得因阈值尚未配置而绕过审批或形成空白流程。

## 9. 手机端重点

- 一笔资金一张卡；
- 支持拍照、上传凭证、自动保存草稿；
- 金额、账户、日期使用大输入控件；
- 退回后直接进入缺失字段。

## 10. 飞书实机验收

```text
EMP004 真实身份登录
→ 新建一笔真实收款或付款
→ 上传凭证并保存草稿
→ 重新进入继续填写
→ 提交张敬东复核
→ 查看结果
```

## 11. V2 预留

银行流水匹配、自动凭证归档、账户异常提醒和现金流预测。

# 4.5 EMP005 石昊昕｜行政总监

## 1. 岗位定位

石昊昕承担行政采购报销和照护师工资决算两个独立职责。首页和页面必须分区，不能把采购报销与照护师工资混成一套表单。

## 2. 唯一核心目标

```text
行政采购：上传凭证、确认识别结果、提交财务
照护师工资：核对服务数据、生成工资候选、提交审批
```

## 3. 首页第一眼看什么

**行政采购区：**待上传凭证、待确认识别、待修改、待提交、被退回、付款状态。
**照护师工资区：**本月服务记录、服务天数异常、工资候选、待提交、审批和付款状态。

## 4. 一级菜单

```text
首页
行政采购报销
照护师工资
提交与结果
我的记录
```

## 5. 可查看内容

```text
→ 接待与签约事实自然进入销售经营数据池
→ 10晓磊查看汇总销售和资金状态
```

客户收款不经过10晓磊审批。刘晶只确认银行或账户中的真实到账，张敬东负责复核；销售提交的付款信息本身不等于到账事实。

# 六、V1 数据沉淀范围与阶段定位

## 6.1 数据池阶段定位

五大数据池主要服务于 OMS V2。V1 当前不开发复杂的数据池分析、预测、推荐、经营模型或智能决策功能。

V1 只负责：

- 在真实业务流程中自然产生数据；
- 保存原始业务数据；
- 保留必要分类、状态和来源；
- 保留人工修改记录；
- 为未来 V2 提供可用的数据沉淀。

统一原则：

> V1 负责自然沉淀数据，不额外增加员工录入负担；V2 再基于沉淀数据建设分析、预测和智能经营能力。

## 6.2 五大长期基础数据池

五大长期基础数据池固定为以下五项，不增加、不拆分：

| 数据池 | V1自然来源 | V1当前处理 | V2预留价值 |
|---|---|---|---|
| 食材采购数据池 | 薛子渝在食材采购流程中产生的采购单、凭证、明细和到货事实 | 保存原始内容、必要分类、审核状态、付款关联和人工修正 | 价格趋势、供应商比价、采购频率和成本分析 |
| 行政采购数据池 | 石昊昕在行政采购与报销流程中产生的凭证、物品或服务信息 | 保存原始内容、必要分类、审核状态、付款关联和人工修正 | 采购周期、消耗趋势、预算和供应商分析 |
| 照护师数据池 | 实际服务关联、服务时间、工资决算、审批、付款和核算事实 | 保存基础身份、实际服务依据和结算历史 | 服务次数、累计天数、空档期、稳定性和成本分析 |
| 出馆客户资料库 | 尚丽娜维护的必要在住资料、出馆前补全资料及刘芳羽确认的出馆事实 | 保存原始客户资料、来源、状态、版本和必要标签 | 老客户管理、二次营销、转介绍和客户画像 |
| 销售经营数据池 | 杨欢欢、薛子渝、刘芳羽分别产生的本人接待、签约、合同和付款申报事实 | 保存原始销售事件、归属、状态、来源和财务确认结果 | 渠道效果、转化率、个人表现和销售趋势 |

说明：

- 三名销售分别维护本人销售事实，系统自动汇总，不要求任何一名销售人工填写全体汇总数字。
- 食材采购由薛子渝实际录入与执行；周志朋保持纯只读，不创建采购需求或确认采购订单。
- 出馆客户资料由尚丽娜按最少字段维护，已有资料必须自动带出，不得重复录入。
- 照护师池只自然接收实际服务和结算业务产生的数据，V1不建设复杂排班、匹配或质量评分。

## 6.3 V1 日常业务数据

以下数据在 OMS V1 中正常保存和使用，但不并列为新的基础数据池：

### 财务业务数据

- 实际收款；
- 实际付款；
- 凭证；
- 待收款与待付款；
- 账户日结；
- 会计复核与核算结果。

财务业务数据可以为食材采购、行政采购、照护师和销售经营数据池提供付款、到账和核算依据，但不属于新的长期基础数据池。

### 运营业务数据

- 当前在住；
- 实际入住与出馆；
- 当前房态；
- 未来排房；
- 运营确认状态。

运营业务数据可以为照护师、出馆客户资料库和销售经营数据池提供实际入住、房间和出馆依据，但不属于新的长期基础数据池。

### 薪资业务数据

- 正式员工工资候选、审批、付款和核算；
- 照护师工资候选、审批、付款和核算。

薪资业务数据中的照护师服务与结算事实可以关联照护师数据池，但正式员工工资不因此形成新的长期基础数据池。

## 6.4 V1沉淀边界

- V1只保存真实业务过程中自然形成的数据，不额外要求员工为数据池重复填写；
- 统计项由系统根据原始事件自动计算，不作为人工录入字段；
- 人工修改必须保留原值、新值、修改人、时间和原因；
- 历史资料可按可靠来源补录，但不得冒充当前事实；
- 数据池的复杂查询、趋势、预测、推荐和智能决策统一进入V2。

# 七、共用移动端规则

1. 手机端不得使用桌面侧栏缩小版。
2. 首页只显示当前岗位最重要的信息和待办。
3. 一级菜单进入独立二级页面，不在首页无限堆叠。
4. 录入岗位支持拍照、自动识别、草稿保存和退回补充。
5. 审核岗位使用大按钮，查看与执行明确分开。
6. 纯只读岗位不得显示任何写入按钮。
7. 表单单列，一页处理一个任务，字段标签在输入框上方。
8. 除“OMS”外，员工可见界面全部中文。
9. 所有业务数据标明状态和截止时间。
10. 返回路径清晰，不要求横屏或缩放。

# 十二、当前结论

本版统一标准：

```text
一个岗位一个核心目标
能自动就不手工
能少录就不多录
录入、确认、复核、审批相互分离
V1 先真实使用
V1 自然沉淀数据
V2 再做复杂和智能
```
