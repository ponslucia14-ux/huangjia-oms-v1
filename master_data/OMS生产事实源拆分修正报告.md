## 一、修正目标

四份 Excel 已完成结构化消化。本次修正的核心是生产事实源边界：

```text
sales.json    -> 销售事实
finance.json  -> 财务事实
room.json     -> 房间与房态事实
stay.json     -> 入住、待入住、出馆与入住计划
customer.json -> 客户基础身份
contract.json -> 合同、套餐与合同计划金额
```

`room.json` 不再承载客户、合同、入住计划。

## 三、金额口径说明

### 1. 销售总额 vs 合同计划金额

| 指标 | 金额 |
|---|---:|
| 销售总额 | 4,686,582.00 |
| 合同计划金额 | 3,715,199.00 |
| 差异 | 971,383.00 |

口径差异：

- 销售总额来自销售明细 Excel，范围是 2026 年销售明细，包含内店、外店、餐食等销售事实。
- 合同计划金额来自签约客户一览 Excel，本轮只导入 2026.06-2026.12 的当前和未来入住计划。
- 两者不是同一统计范围，不能直接要求相等。

### 2. 实际到账 vs 财务收入

| 指标 | 金额 |
|---|---:|
| 销售实际到账 | 3,956,440.00 |
| 财务收入 | 7,682,746.66 |
| 差异 | 3,726,306.66 |

口径差异：

- 销售实际到账来自销售明细，是销售合同维度的到账金额。
- 财务收入来自财务月报，是财务流水维度，包含多类收入、跨月收款、转入、餐食、押金/尾款等。
- 两者一个是销售合同口径，一个是财务流水口径，不应直接相等。

## 五、数据冲突与异常清单

| 类型 | 数量 | 处理 |
|---|---:|---|
| 房态标记缺房号 | 2 | 保留异常，不猜测补齐 |
| 房态入住标记与签约计划姓名重叠 | 23 | 属正常交叉验证候选，不自动合并 |
| 销售中非 OMS 正式销售归属 | 2 | 保留 warning，不自动绑定 |
| 照护师独立事实源 | 0 | 当前四份 Excel 不建设独立照护师模块 |

说明：

- 房态重叠代表同一客户可能同时出现在房态与签约计划中。
- 当前不按姓名自动合并，因为 OMS 正式引用应使用稳定实体 ID。
- 两条缺房号记录继续作为异常保留。

## 六、照护师口径

当前四份 Excel 不提供独立照护师资源池。

现阶段规则：

- 服务对应关系从真实在住/排房数据读取。
- 照护师工资从财务或工资决算数据读取。
- 不为了凑模块新建 caregiver 数据源。
- 照护师资源池留待未来数据积累后建设。
